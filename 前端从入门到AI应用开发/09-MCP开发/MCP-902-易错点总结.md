# 易错点总结（MCP-902）

## 🎯 本节学习目标

- 复盘 MCP 开发中的 schema、权限、返回格式和调试问题。
- 理解它在 MCP Client、Server、Tool、Resource 或 Prompt 中的位置。
- 能够把本节知识应用到 AI Agent 连接业务数据和自动化工具的场景中。

---

## 📖 什么是易错点总结

易错点总结 是 MCP 开发中的关键知识点。MCP 的目标不是让模型“知道一切”，而是用清晰、受控的协议让 AI Agent 在需要时访问外部工具、资源和提示模板。

在真实项目中，MCP 常用于连接本地文件、知识库、业务系统、自动化脚本和内部数据。它的价值在于把能力边界、输入输出和权限约束显式化。

学习本节时，要同时关注模型体验和工程安全：工具描述要让模型会用，权限边界要让系统可控。

---

## 🧠 原理讲解

MCP 的基本调用链路可以简化为：

```text
AI Host
  ↓
MCP Client
  ↓ 能力发现 / 工具调用 / 资源读取
MCP Server
  ↓
本地文件、数据库、API 或脚本
```

本节核心入口：

```ts
const mistakes = ["unclear tool schema"]
```

无论是 Tool、Resource 还是 Prompt，都要把“模型能看到什么、能传什么、能得到什么”设计清楚。

## ✅ 完整代码

下面用一个 MCP Tool 风格的函数演示 易错点总结 的基本结构：

```ts
type ToolInput = {
  topic: string;
  detailLevel?: "brief" | "normal" | "deep";
};

type ToolResult = {
  content: Array<{
    type: "text";
    text: string;
  }>;
};

function validateInput(input: ToolInput): void {
  if (!input.topic.trim()) {
    throw new Error("topic 不能为空");
  }
}

async function callTool(input: ToolInput): Promise<ToolResult> {
  validateInput(input);

  return {
    content: [
      {
        type: "text",
        text: `主题：${input.topic}\n场景：帮助 MCP Server 更稳定地被 Agent 调用\n核心：const mistakes = ["unclear tool schema"]`
      }
    ]
  };
}
```

---

## 🔍 逐行解析

`type ToolInput` 定义工具可接受的输入参数。

`detailLevel?` 表示可选参数，调用方可以不传。

`type ToolResult` 定义工具返回给模型的内容结构。

`validateInput(input)` 在执行工具前做基础校验。

`throw new Error(...)` 用明确错误阻止无效调用继续执行。

`callTool(input)` 模拟一次工具调用，并返回模型可继续阅读的文本内容。

`content` 数组让返回值可以承载一段或多段结果。

---

## 🌐 浏览器表现

MCP Server 本身通常不直接渲染浏览器页面，但它会影响 AI 客户端中的交互体验。

当工具描述清晰、输入校验明确、返回内容结构稳定时，AI Agent 更容易正确调用工具，并把结果解释给用户。

## 📦 常见属性 / API

| 属性/API | 类型 | 说明 | 示例 |
|---------|------|------|------|
| `const mistakes = ["unclear tool schema"]` | MCP 概念/API | 本节核心入口 | `const mistakes = ["unclear tool schema"]` |
| `Tool` | MCP 能力 | 暴露可执行动作 | 生成标题、检索笔记 |
| `Resource` | MCP 能力 | 暴露可读取上下文 | `docs://article/001` |
| `Prompt` | MCP 能力 | 暴露可复用提示模板 | 内容复盘 Prompt |
| `schema` | 输入约束 | 描述工具参数结构 | `topic: string` |
| `content` | 返回字段 | 返回给模型的内容数组 | `{ type: "text" }` |

---

## ⭐⭐⭐⭐⭐ 必学重点

- MCP Server 暴露的是受控能力，不是无限权限。
- Tool 设计要让模型知道什么时候用、怎么传参、结果代表什么。
- 易错点总结 要同时考虑可用性、权限、安全和可调试性。

---

## ⚠️ 易错点

- 易错点1：工具描述太模糊，模型不知道何时调用。正确做法：描述输入、输出和适用场景。
- 易错点2：没有输入校验。正确做法：所有 Tool 参数都要校验。
- 易错点3：让工具拥有过大权限。正确做法：最小权限、明确范围、记录审计日志。

---

## 💡 最佳实践

- Tool 名称使用动词短语，例如 `search_notes`、`create_summary`。
- Resource URI 要稳定、可预测、便于调试。
- 出错时返回可理解、可恢复的信息，而不是模糊失败。
